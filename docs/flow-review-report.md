# 采购→出货全链路排查报告

> 排查范围：前端页面 + 后端路由 + 状态机定义
> 日期：2026-05-07

---

## 一、端到端流程总览

```
采购端：
  采购单(草稿) → 采购单(已确认) → [创建收货单] → PDA收货 → PDA上架 → 入库完成 → 采购单(已完成)

销售端：
  销售单(草稿) → 预占库存 → 发起出库 → [仓库任务创建]
                    ↓
         拣货中 → 待分拣 → 待复核 → 待打包 → 待出库 → 已出库
```

---

## 二、发现的问题

### 问题 1：采购确认→收货需要手动跳转

**现状**：采购单确认后，用户需要自己找到"收货订单"菜单，手动创建收货单。采购单页面没有引导。

**影响**：用户不清楚下一步该做什么。采购单状态变成"已确认"后就停在那里了。

**建议**：在采购单详情页加一个"创建收货单"按钮，点击后自动带入采购单信息跳到收货创建页。类似于销售单创建后跳转预占库存的流程。

---

### 问题 2："预占库存"和"发起出库"两步太多余

**现状**：销售单创建后，用户需要先点"预占库存"，再点"发起出库"。两步之间不能跳过。

**后端状态定义**：

```
草稿(1) → 已占库(2) → 拣货中(3) → 已出库(4)
```

实际上"预占库存"是后端做库存锁定的动作，"发起出库"是创建仓库任务。但对用户来说，我的诉求是"把这个单子发出去"。

**建议**：合并为一个按钮"确认出库"，内部依次执行预占 + 发起出库。用户不需要知道中间有预占这个步骤。

---

### 问题 3："已占库"命名不直观

**现状**：销售单状态名称 "已占库" 是直译 "reserved/occupied"。用户（尤其是一线操作员）可能不理解。

**对比**：

| 当前 | 建议 |
|------|------|
| 已占库 | 已确认 |
| 拣货中(销售单) | 履约中 |
| 已出库(销售单) | 已完成 |

"已占库"改成"已确认"后，销售单的流程变为：

```
草稿(1) → 已确认(2) → 履约中(3) → 已完成(4) → 已取消(5)
```

这组命名不需要理解"预占"概念，只看字面就知道什么意思。

---

### 问题 4：仓库任务 6 个状态偏多

**现状**：仓库任务状态链：

```
拣货中(2) → 待分拣(3) → 待复核(4) → 待打包(5) → 待出库(6) → 已出库(7)
```

**问题**：每个步骤都有独立状态，但实际中"待分拣"和"待复核"有时是同一个操作员在同一时间完成的，区分这两步没有业务意义。

**建议**：合并为 4 个状态：

| 当前 | 精简后 |
|------|--------|
| 拣货中(2) | 拣货中 |
| 待分拣(3) → 待复核(4) | 待核验 |
| 待打包(5) → 待出库(6) | 待出库 |
| 已出库(7) | 已出库 |

或者更激进一点，去掉"待分拣"和"待复核"的区分，改为：

```
拣货中 → 待核验 → 待出库 → 已出库
```

---

### 问题 5：采购单状态 "已完成" 的定义不清晰

**现状**：采购单所有明细收货完毕后，状态变为"已完成"。但用户不知道：
- 是不是所有商品都到齐了？
- 为什么采购单还是"已确认"？

**建议**：在采购单详情页顶部加一个进度条：
```
收货进度: [████████░░] 8/10 件已入库
```

同时在采购单列表中加一列"收货进度"，让用户一目了然。

---

### 问题 6：销售单"履约"信息展示太隐晦

**现状**：销售单列表有一个"履约"徽标，显示 "履约：拣货中"。但这个信息夹在表格里，不够醒目。

**建议**：在销售单详情页加一个简单的卡片展示履约状态链，类似：
```
┌──────────────────────────────────────┐
│  履约进度                             │
│  拣货中 → 待分拣 → 待复核 → 待出库    │
│  ● 已完成  ○ 待处理  ○ 待处理  ○ 待处理│
│                                       │
│  当前在：XXX 库位  操作人：王师傅      │
└──────────────────────────────────────┘
```

## 三、建议优先级

| 优先级 | 改动 | 影响范围 | 风险 |
|--------|------|----------|------|
| P1 | 合并且重命名销售单状态 "已占库"→"已确认"、"拣货中"→"履约中" | 前端页面 + 后端常量 | 低（只改显示名，不改状态值） |
| P1 | 采购单列表加"收货进度"列 | 前端页面 | 低 |
| P2 | 采购单确认后引导创建收货单 | 前端页面 | 低 |
| P2 | 销售单详情页加履约进度卡片 | 前端页面 | 低 |
| P3 | 合并"预占库存"+"发起出库"为一个按钮 | 前端 + 后端 | 中（涉及业务逻辑） |
| P3 | 仓库任务状态从 6 个精简到 4 个 | 后端 + PDA + 前端 | 高（涉及多处） |

## 四、不改现有数据结构的方案

以上所有建议中，P1 和 P2 只涉及：
- 改前端显示文本（状态名）
- 加前端组件（进度条、卡片）
- 不改数据库、不改接口协议、不改后端逻辑

P3 涉及后端逻辑，建议先放一放。
