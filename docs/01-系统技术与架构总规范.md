## 1. 项目总体架构说明

### 1.1 前端架构

| 层级 | 技术选型 | 用途 |
|------|----------|------|
| 构建工具 | Vite | 开发服务器与生产打包 |
| UI 框架 | React 18 | 组件渲染与状态管理 |
| 样式系统 | Tailwind CSS + shadcn/ui | 统一视觉与组件规范 |
| 全局状态 | Zustand | 跨组件共享状态（用户信息、权限、全局配置） |
| 服务端状态 | React Query (TanStack Query) | 接口请求、缓存、同步 |
| 路由 | React Router v6 | 页面路由与权限守卫 |

**架构原则：**
- 所有接口调用必须通过 React Query 管理，禁止在组件内直接使用 `fetch` 或 `axios`。
- Zustand 仅存储全局持久状态，不存储接口响应数据。
- 组件必须分层：Page 层、Container 层、UI 组件层，不允许混写。

---

### 1.2 后端架构

| 层级 | 技术选型 | 用途 |
|------|----------|------|
| 运行时 | Node.js 20+ | 服务运行环境 |
| Web 框架 | Express.js | HTTP 路由与中间件管理 |
| 数据库驱动 | mysql2/promise | MySQL 连接与 SQL 执行 |
| 认证机制 | JWT (jsonwebtoken) | 无状态身份验证 |
| 参数校验 | zod | 接口入参校验 |
| 密码加密 | bcryptjs | 用户密码哈希存储 |

**架构原则：**
- 后端严格分层：`routes → controllers → services → db`。
- 每一层只做本层职责，不得跨层调用或混写逻辑。
- 数据库操作统一封装在 service 层，controller 不得直接执行 SQL。

---

### 1.3 桌面端架构

| 层级 | 技术选型 | 用途 |
|------|----------|------|
| 壳体 | Electron | 桌面应用容器 |
| 渲染进程 | 前端 React 构建产物 | UI 渲染 |
| 主进程 | Electron Main Process | 系统级操作（文件、通知、更新） |
| 自动更新 | electron-updater | 增量更新与版本管理 |

**架构原则：**
- 主进程与渲染进程通信必须通过 `ipcMain` / `ipcRenderer` 严格分离，禁止在渲染进程直接使用 Node 模块。
- 桌面自动更新优先使用正式代码签名；在正式证书未补齐前，必须以 GitHub Actions 固定构建链、GitHub Release 正式产物与服务器哈希校验作为发布基线。
- 桌面端不承载独立业务逻辑，业务逻辑统一在后端 API 处理。

---

### 1.4 PDA 端架构

| 层级 | 技术选型 | 用途 |
|------|----------|------|
| 平台 | Android App | 移动端操作入口 |
| 接口对接 | RESTful API | 与后端统一接口规范对接 |
| 认证 | JWT Token | 与 Web 端共用认证体系 |

**架构原则：**
- PDA 端不存在独立的业务逻辑服务，所有数据操作通过后端 API 完成。
- PDA 使用与 Web 端相同的接口规范与响应格式。
- PDA 的接口权限与 Web 端角色权限体系保持一致。

## 2. 目录结构规范

### 2.1 frontend 目录分层

```
frontend/
├── src/
│   ├── api/              # 所有 API 请求函数（与 React Query 配合）
│   ├── assets/           # 静态资源（图片、图标）
│   ├── components/       # 全局复用 UI 组件
│   │   ├── ui/           # shadcn/ui 组件（不得手动修改）
│   │   └── shared/       # 项目级公共组件（PageHeader、DataTable 等）
│   ├── hooks/            # 自定义 Hook（含 React Query hooks）
│   ├── layouts/          # 页面布局框架
│   ├── lib/              # 工具函数、常量、类型定义
│   ├── pages/            # 页面级组件（每个模块一个子目录）
│   │   └── [module]/
│   │       ├── index.tsx        # 列表页
│   │       ├── detail.tsx       # 详情/编辑页
│   │       └── components/      # 该模块私有组件
│   ├── router/           # 路由配置与权限守卫
│   ├── store/            # Zustand 全局状态
│   └── types/            # 全局 TypeScript 类型定义
```

**规则：**
- `pages/` 下的组件只负责页面编排，业务请求逻辑封装在 `hooks/`。
- `components/ui/` 为 shadcn/ui 自动生成目录，禁止手动修改其内文件。
- `lib/` 中的函数必须是纯函数，不得引入 React 依赖。

---

### 2.2 backend 目录分层

```
backend/
├── src/
│   ├── config/           # 数据库连接、环境变量、常量配置
│   ├── middleware/       # 全局中间件（认证、错误处理、日志）
│   ├── modules/          # 业务模块（每个模块独立目录）
│   │   └── [module]/
│   │       ├── [module].routes.js
│   │       ├── [module].controller.js
│   │       └── [module].service.js
│   ├── utils/            # 工具函数（响应封装、加密、时间处理等）
│   └── app.js            # Express 应用入口（注册路由与中间件）
├── index.js              # 服务启动入口
```

---

### 2.3 routes / controllers / services 分层职责

| 层 | 职责 | 禁止事项 |
|----|------|----------|
| **routes** | 注册路由路径、挂载中间件、调用 controller | 禁止写任何业务逻辑或 SQL |
| **controller** | 解析请求参数、调用 service、返回响应 | 禁止直接操作数据库、禁止写复杂业务逻辑 |
| **service** | 执行业务逻辑、操作数据库、处理事务 | 禁止直接操作 HTTP 请求/响应对象 |

**示例：正确调用链**

```
POST /api/inventory/items
  → routes: 校验 JWT → 调用 controller.createItem
  → controller: 提取 req.body → 调用 service.createItem → res.json(result)
  → service: 校验业务规则 → 执行 SQL → 返回数据
```

## 3. 接口设计统一规范

### 3.1 RESTful 规则

| 操作 | 方法 | 路径示例 |
|------|------|----------|
| 获取列表 | `GET` | `/api/inventory/items` |
| 获取单条 | `GET` | `/api/inventory/items/:id` |
| 创建 | `POST` | `/api/inventory/items` |
| 更新 | `PUT` | `/api/inventory/items/:id` |
| 删除（逻辑） | `DELETE` | `/api/inventory/items/:id` |

**规则：**
- 路径全小写，单词间用 `-` 分隔，不使用驼峰。
- 资源名称使用复数名词。
- 不在路径中嵌套超过两层资源（`/api/orders/:id/items` 为上限）。

---

### 3.2 统一响应格式

所有接口响应必须使用以下结构，无论成功或失败：

```json
{
  "success": true,
  "message": "操作成功",
  "data": {}
}
```

| 字段 | 类型 | 说明 |
|------|------|------|
| `success` | Boolean | 业务层面的成功/失败标识 |
| `message` | String | 人类可读的结果描述，中文 |
| `data` | Object / Array / null | 响应数据体，失败时为 `null` |

列表接口在 `data` 中必须附带分页信息：

```json
{
  "success": true,
  "message": "查询成功",
  "data": {
    "list": [],
    "pagination": {
      "page": 1,
      "pageSize": 20,
      "total": 100
    }
  }
}
```

---

### 3.3 错误处理规则

- 所有 controller 必须使用 `try/catch`，catch 中将错误传递给 `next(err)`。
- 全局错误处理中间件统一捕获，格式化后返回响应，禁止在 controller 内自行构建错误响应。
- 业务错误（如库存不足）抛出自定义 `AppError`，HTTP 错误与业务错误分离处理。

```js
// utils/AppError.js
class AppError extends Error {
  constructor(message, statusCode = 400) {
    super(message);
    this.statusCode = statusCode;
    this.isOperational = true;
  }
}
```

---

### 3.4 HTTP 状态码规范

| 状态码 | 场景 |
|--------|------|
| `200` | 请求成功（含查询、更新、删除） |
| `201` | 资源创建成功 |
| `400` | 请求参数错误、业务规则不满足 |
| `401` | 未登录或 Token 无效 |
| `403` | 已登录但无权限 |
| `404` | 资源不存在 |
| `500` | 服务端未预期错误 |

## 4. 数据库设计规范

### 4.1 表命名规则

- 全小写，单词间使用 `_` 分隔。
- 使用模块前缀区分业务域，格式：`[模块]_[资源]`。
- 示例：`inventory_items`、`order_headers`、`user_roles`、`sys_configs`。
- 禁止使用保留字命名（如 `order`、`user`），需加前缀。

---

### 4.2 字段命名规则

- 全小写下划线风格（snake_case）。
- 布尔字段使用 `is_` 前缀，例：`is_active`、`is_deleted`。
- 外键字段命名为 `[关联表单数]_id`，例：`warehouse_id`、`supplier_id`。
- 禁止使用单字母字段名，禁止使用中文字段名。

---

### 4.3 必备字段

所有业务表必须包含以下字段：

| 字段 | 类型 | 说明 |
|------|------|------|
| `id` | `BIGINT UNSIGNED AUTO_INCREMENT PRIMARY KEY` | 主键 |
| `created_at` | `DATETIME DEFAULT CURRENT_TIMESTAMP` | 创建时间 |
| `updated_at` | `DATETIME DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP` | 更新时间 |

---

### 4.4 逻辑删除规范

- 所有业务数据表使用逻辑删除，不得物理删除行数据。
- 逻辑删除字段：`deleted_at DATETIME DEFAULT NULL`（为 `NULL` 表示未删除）。
- 所有查询必须默认加上 `WHERE deleted_at IS NULL` 条件。
- 系统日志表、操作记录表等只追加不删除，无需 `deleted_at` 字段。
- 涉及唯一约束的字段（如单据编号），唯一索引需联合 `deleted_at` 或改用软删除安全策略。

---

## 5. 代码编写规则

### 5.1 统一 CRUD 工厂

所有标准 CRUD 操作必须通过统一工厂函数生成，不允许各模块自行重复实现相同结构。

工厂函数提供以下标准方法：
- `findAll({ page, pageSize, filters })` — 带分页与筛选的列表查询
- `findById(id)` — 单条查询
- `create(data)` — 新增
- `update(id, data)` — 更新
- `softDelete(id)` — 逻辑删除

如业务有特殊逻辑，在模块自身的 service 中继承或扩展，不得修改工厂函数本体。

---

### 5.2 统一错误处理中间件

后端必须注册全局错误处理中间件，位于 `middleware/errorHandler.js`，统一处理：
- `AppError`（业务异常）→ 返回对应 `statusCode` 与 `message`
- MySQL 错误（`ER_DUP_ENTRY` 等）→ 转换为可读业务错误
- 未知错误 → 返回 `500`，不暴露堆栈给客户端

---

### 5.3 禁止事项（强制）

| 禁止行为 | 说明 |
|----------|------|
| 在 routes 中写业务逻辑 | routes 只声明路径与挂载 controller |
| 在 controller 中写 SQL | SQL 只允许在 service 层出现 |
| 在 controller 中写复杂条件判断 | 业务规则必须在 service 层处理 |
| AI 随意变更代码写法 | 所有模块必须遵循既有结构模板 |
| 跳过错误处理直接 `res.json` | 必须经过统一错误链路 |
| 硬编码配置值 | 所有配置必须通过 `config/` 或环境变量提供 |

---

### 5.4 AI 辅助开发约束

- AI 工具（包括 Cursor、Copilot 等）生成的代码必须符合本规范，不得因为"风格差异"绕过规范。
- AI 不得批量生成与现有模块结构不一致的代码。
- AI 生成新模块时，必须以现有成熟模块为模板，逐步生成，不得一次性生成整个模块的全量代码后提交审查。

---

## 6. 安全规范

### 6.1 JWT 校验规则

- 所有需认证的接口必须挂载 `authMiddleware`，位于 `middleware/auth.js`。
- Token 存储在客户端 `localStorage`，请求时通过 `Authorization: Bearer <token>` 头部传递。
- Token 有效期：访问 Token 7 天，刷新 Token 根据业务需求配置。
- Token 中只存储最小必要信息：`{ userId, roleId, iat, exp }`，不存储敏感字段。
- 服务端不维护 Token 黑名单（无状态），注销通过客户端删除 Token 实现。

---

### 6.2 接口权限分层

| 等级 | 说明 | 示例 |
|------|------|------|
| 公开接口 | 无需 Token | 登录、健康检查 |
| 认证接口 | 需要有效 Token | 所有业务接口 |
| 权限接口 | 需要 Token + 特定角色/权限 | 删除、审批、系统配置 |

- 权限粒度到接口级别，通过 `permissionMiddleware(permission)` 中间件控制。
- 权限码格式：`[模块]:[动作]`，例：`inventory:delete`、`order:approve`。

---

### 6.3 参数校验规则

- 所有 `POST` / `PUT` 接口必须在 routes 层挂载参数校验中间件，校验通过后才进入 controller。
- 使用 zod 定义 schema，校验失败直接返回 `400`。
- 禁止在 controller 或 service 中做 "临时" 参数校验（if 判断），所有校验在进入 controller 前完成。
- SQL 参数必须使用占位符（`?`），严禁拼接字符串构造 SQL。

---

## 7. 前端 UI 规范

### 7.1 组件库规范

- 全项目统一使用 shadcn/ui，不引入其他 UI 库（禁止混用 antd、MUI、Element Plus 等）。
- shadcn/ui 组件通过 CLI 安装到 `components/ui/`，不手动创建 UI 基础组件。
- 需要定制样式时，在 `components/shared/` 中封装基于 shadcn/ui 的业务组件，不修改 `components/ui/` 中的源文件。

---

### 7.2 颜色系统

- 所有颜色引用必须使用 Tailwind CSS 语义变量（如 `bg-primary`、`text-muted-foreground`），禁止硬编码 HEX 或 RGB 值。
- 主题色在 `tailwind.config.js` 与 CSS 变量中统一定义，不允许在组件内单独修改。
- 深色模式支持通过 CSS 变量切换实现，不得使用 `dark:` 前缀在组件中写双套样式。

---

### 7.3 PageHeader 规范

所有页面必须使用统一的 `PageHeader` 组件，结构如下：

```tsx
<PageHeader
  title="页面标题"
  description="可选描述"
  actions={<Button>操作按钮</Button>}
/>
```

- `title` 必填，使用中文，简洁准确。
- `actions` 区域放置该页面的主要操作按钮（新增、导出等）。
- 禁止在页面内自行编写标题 + 操作按钮的 flex 布局，统一使用 `PageHeader`。

---

### 7.4 页面可复用性要求

- 所有列表页必须支持：搜索筛选、分页、列表/表格切换（视需求）。
- 所有表单页（新增/编辑）使用同一页面组件，通过 `id` 参数区分新增与编辑模式。
- 弹窗使用 shadcn/ui `Dialog`，抽屉使用 `Sheet`，选择弹窗使用 `Combobox` 或 `Command`。
- 表格使用统一封装的 `DataTable` 组件，不得为每个模块单独实现表格逻辑。

---

## 8. 开发流程规范

### 8.1 新模块开发步骤

新增业务模块必须按以下顺序推进，不得跳步：

1. **数据库设计** — 设计表结构，写 migration 脚本，评审通过后执行
2. **后端接口** — 按照模板创建 routes / controller / service，完成接口联调
3. **前端 API 层** — 在 `api/` 目录编写对应请求函数与 React Query hooks
4. **前端页面** — 搭建列表页、表单页，接入真实接口
5. **权限配置** — 在权限系统中注册该模块的权限码
6. **测试验证** — 主流程手动测试通过

---

### 8.2 代码提交规则

提交信息格式（Conventional Commits）：

```
<type>(<scope>): <subject>

type: feat | fix | refactor | chore | docs | style | test
scope: 模块名称，如 inventory、order、auth
subject: 中文描述，不超过 50 字
```

示例：
```
feat(inventory): 新增库存盘点模块列表页与接口
fix(auth): 修复 Token 过期后未跳转登录页的问题
refactor(order): 将订单状态计算逻辑抽离至 service 层
```

**规则：**
- 禁止提交包含 `console.log`、调试代码的内容到主分支。
- 禁止将 `.env` 文件提交到版本控制。
- 每次提交只包含一个完整的逻辑变更，不得将多个模块的改动混入同一次提交。

---

### 8.3 禁止 AI 批量生成无结构代码

- AI 工具不得在未阅读现有同类模块代码的情况下，生成新模块的全量代码。
- AI 生成代码前必须先读取：
  - 现有同类模块的 routes / controller / service 文件
  - 本规范文档
- AI 不得擅自引入未在本规范中列出的新依赖包。
- AI 对现有代码的重构必须经过人工确认，不得直接覆盖现有实现。

---

*本文档为强制性规范，优先级高于任何 AI 工具的默认行为与个人编码偏好。*
