# 极序 Flow 系统未来蓝图（现实修订版）

> 版本：v1.13
> 状态：生效中  
> 最后更新：2026-04-12

本文档用于把《极序 Flow 系统未来蓝图（非 AI 版）》收敛成**当前系统可执行、可上线、可迭代**的版本。目标不是减少野心，而是把未来蓝图改成符合当前代码现实、业务现实与实施节奏的路线图。

---

## 1. 修订目标

未来蓝图不再按“所有模块同时铺开”的方式推进，而是改成三层路线：

1. **先固化核心闭环**
   - 主数据、采购/销售、库存、仓库作业、打印、审计
2. **再补管理能力**
   - 首页工作台、对账、报表、审批、提醒
3. **最后扩展外围模块**
   - 售后、品牌系列、复杂价格策略、物流追踪、首页个性化

本修订版遵循以下原则：

- 蓝图必须服务于当前 ERP + WMS 主闭环，而不是追求模块数量。
- 任何新模块都必须复用现有标准页模板、打印中心、日志和异常处理能力。
- 术语、条码、单据关系必须与当前系统一致，不允许“文档写一套，系统跑一套”。

- ERP 主流程已经具备：

  - 库存
  - 调拨
  - 波次

  - 异常工作台

- 系统统一能力已经初步建立：
  - 统一表格
  - 统一页面模板
  - 条码体系
  - 桌面端与 PDA 端协同

### 2.2 当前能力成熟度分级

为避免“已具备”被误解为“已成熟”，当前能力按如下分级理解：

- **已稳定**
  - 采购、销售、收货订单、库存、打印任务中心、桌面端更新
- **已上线但需持续优化**
  - PDA 作业链路、异常工作台、跨网络打印、统一表格与页面模板
- **已有雏形但未成熟**
  - 报表、首页工作台、财务口径、审批流基础能力

### 2.2.1 当前已落地的强化项

以下能力已经进入当前代码主线，可视为 Phase 1 的已实施内容：

- 商品主数据关键字段硬校验已前移：

  - 取消占库

  - 异常工作台与通知中心可直接定位到具体收货订单详情

- Phase 2 管理增强已开始落地：
  - 岗位工作台基础版已按仓库、销售/客服、管理三类岗位聚合待办
  - 对账基础版已完成客户/供应商双口径查询、筛选、导出与原单回跳

  - 审批与提醒基础版已接入岗位工作台、异常工作台与通知中心

  - 对账页已开始从“查询页”向“核对页”收口，默认优先展示待核对记录
  - 审批与提醒顶部已优先展示财务 / 系统级提醒，不再复用岗位工作台的操作型最优先待办
  - 岗位工作台已补入“今日处理顺序”，便于仓库、销售客服、管理角色按固定顺序处理日常待办
  - 收货订单详情已补入审核退回原因必填、补录动作可见化、重新审核入口、打印批次视图与补打结果回写
  - 通知中心、异常工作台、收货详情三者已统一为“收货异常闭环”，从提醒入口可直接落到详情页对应处理区块
  - 波次详情、打印查询、通知中心、异常工作台已开始按统一异常编码处理出库打印失败与波次推进异常
  - 仓库任务详情、PDA 打包页、PDA 出库页已按统一闭环提示展示待打包、箱贴打印、待出库、出库确认阶段
  - 物流标签打印异常已进入通知中心、异常工作台、打印查询、PDA 出库页、仓库任务详情的统一闭环
  - 页面烟雾门禁已覆盖收货详情、仓库任务、波次详情、入库打印查询、出库打印查询、物流标签打印查询等主链入口
  - 当前优先级仍保持为“先收 P1 主链，再做 P2 管理增强”

### 2.3 当前仍未完全成熟的能力

- 财务口径仍未完全闭环
- 审批流仍不适合做复杂规则设计器，当前仅停留在基础待办聚合
- 首页岗位工作台已完成基础版，且已按固定优先级展示待办，但仍需要结合实际岗位继续细化入口权重和排序
- 售后链路尚未形成完整闭环
- 高级价格体系尚未形成规则引擎
- 品牌 / 系列 / 单位包装等主数据层尚未值得单独扩张

## 3. 分阶段蓝图

## Phase 1：经营闭环（当前主线）

### 目标

把现有主线做成“可长期稳定运营”的骨架，确保任何关键业务动作都有来源、去向、日志、打印与异常处理闭环。

### 范围

- 商品
- 客户
- 供应商
- 仓库 / 货架 / 货位
- 采购
- 销售
- 收货订单
- 库存
- 仓库作业
- 打印中心
- 审计 / 日志 / 异常 / 提醒

### 分段实施建议

- **P1a：主业务闭环**
  - 商品、客户、供应商、仓库主数据
  - 采购、销售、收货订单、库存、PDA 主链路
  - 打印中心与关键条码闭环
- **P1b：稳定运营支撑**
  - 异常工作台
  - 操作日志 / 打印日志
  - 提醒 / 通知
  - 单据编号规则
  - 导出与补打统一

### 进入条件

- 当前系统继续以 ERP + WMS 主闭环为最高优先级
- 不新增会打断主线交付的大型外围模块
- 统一术语、条码、状态流转已成为硬约束

### 退出条件

- 采购、销售、收货、库存、PDA、打印可连续稳定运行
- 关键单据可追踪上游、下游、操作时间线和打印记录
- 关键异常可进入异常工作台或日志链路处理
- 关键页面已统一页面骨架、表格规范和术语

### 本阶段必须成立的验收标准

- 所有核心单据都能看到上游来源、下游结果和操作时间线
- 所有关键打印都进入统一打印任务中心
- 所有异常都能进入异常工作台或日志追踪
- 所有 PDA 作业都与 ERP 术语、条码、打印动作保持一致
- 所有列表页、详情页、单据页继续复用统一骨架

## Phase 2：管理增强

### 目标

在主闭环稳定后，增强经营分析与岗位协同能力。

### 范围

- 首页岗位工作台
- 客户 / 供应商对账
- 利润分析
- 库存分析
- 作业绩效
- 审批流基础版
- 消息提醒基础版
- 导出体系统一化

### 当前推进顺序

1. 岗位工作台基础版
2. 客户 / 供应商对账基础版
3. 利润 / 库存分析基础版
4. 审批流基础版与提醒基础版

### 本阶段输出要求

- 管理能力必须建立在真实业务数据之上
- 先做“基础版”，不直接上复杂规则设计器
- 所有分析页必须支持下钻到原始单据或任务
- 所有报表页上线前必须经过报表烟雾检查，避免再次出现页面可打开但接口 500 的情况

### 进入条件

- Phase 1 已达到稳定运营状态
- 打印、库存、仓库任务口径已基本稳定
- 主线问题处理已不再长期依赖人工补洞

### 退出条件

- 首页工作台、对账、分析、提醒可服务日常管理
- 关键分析页能够下钻回原始单据、任务、打印记录
- 审批和提醒具备基础可用性，但仍保持简化规则

## Phase 3：扩展能力

### 目标

在主线和管理能力稳定后，再做细分场景与高级能力。

### 范围

- 售后工单
- 退换补发
- 复杂价格策略
- 品牌 / 系列
- 单位包装换算
- 物流异常追踪中心
- 首页个性化
- 深度组织岗位与数据边界控制

### 本阶段原则

- 不打断现有闭环
- 不重复造一套页面或状态体系
- 高级能力必须复用前两阶段已经稳定的组件与流程

### 进入条件

- 前两阶段已稳定运行
- 主流程不再频繁因基础问题返工
- 管理能力已经可以支撑日常经营

### 退出条件

- 扩展能力不再破坏主线稳定性
- 高级规则、售后、物流追踪等能力可以按模块独立演进

## 4. 主数据修订原则

### 4.1 先收敛成 5 个核心主对象

当前最应该稳定的主数据只有：

- 商品
- 客户
- 供应商
- 仓库 / 货架 / 货位
- 打印模板 / 打印机绑定

### 4.1.1 一期必须稳定的关键字段

- **商品**
  - SKU
  - 产品条码
  - 进价
  - `A / B / C / D` 四档价格
  - 分类
  - 库存属性
- **客户**
  - 客户编码
  - 价格等级
  - 结算方式
  - 默认地址
  - 联系人
- **供应商**
  - 供应商编码
  - 结算方式
  - 联系人
  - 到货主体信息
- **仓库 / 货架 / 货位**
  - 编码规则
  - 启停状态
  - 所属仓库关系
  - 打印 / 绑定规则
- **打印模板 / 打印机绑定**
  - 模板类型
  - 适用业务
  - 目标打印机
  - 在线状态

### 4.2 这些对象暂不视为一期独立模块

- 品牌与系列
- 单位与包装换算
- 结算方式与物流公司复杂配置

处理原则：

- 一期先作为商品或单据的附属配置能力存在
- 等一线流程稳定后，再决定是否升级为独立模块

## 5. 价格体系修订原则

### 5.1 一期价格模型

当前系统价格体系统一为：

- 商品维护进价
- 系统生成四档客户价：`A / B / C / D`
- 客户绑定价格等级
- 销售单按客户绑定等级自动带价

### 5.2 后续价格能力分层

- **一期**
  - 四档价模型
- **二期**
  - 客户等级价
  - 特殊客户价
- **三期**
  - 活动价
  - 生效时间
  - 报价审批

### 5.2.1 当前阶段例外处理原则

- 默认允许销售单在带出价格后手工改价
- 手工改价必须保留操作日志
- 低于进价的价格必须能被识别，并预留拦截或审批入口
- 当前阶段特殊价优先级高于客户等级价，但不引入完整规则引擎
- 当前阶段不做多套价格体系并行解释

### 5.3 禁止事项

- 当前阶段不直接做复杂价格规则引擎
- 当前阶段不做“多套价格体系并行解释”

---

## 6. 仓库作业与条码术语修订

### 6.1 当前标准仓库作业流程

当前仓库主流程统一为：

1. 收货
2. 打印库存条码
3. 上架
4. 拣货
5. 复核
6. 打包
7. 发货
8. 塑料盒拆分 / 散件作业

### 6.2 统一术语

仓库作业与文档统一使用以下术语：

- 产品条码
- 库存条码
- 塑料盒条码
- 物流条码
- 货架条码
- 波次条码

禁止继续使用旧术语或混合叫法。

### 6.3 收货流程修订

收货入库蓝图统一改为：

- 采购单提交后，不直接生成 PDA 任务
- 收货订单按“供应商一次到货”建单
- 收货订单提交到 PDA 后，PDA 才接收现场任务
- PDA 负责收货、打印库存条码、上架
- 财务审核与仓库动作分离

即：

`采购单 -> 收货订单 -> PDA 收货任务 -> 上架 -> 财务审核`

当前系统已经按这条链路实施，不再保留“采购单直上 PDA”作为正式入口。

### 6.4 ERP 与 PDA 职责边界

- **ERP 负责**
  - 建立采购单
  - 建立收货订单
  - 提交到 PDA
  - 审核
  - 导出
  - 整单 / 明细 / 条码补打
  - 异常处理
  - 审核退回后的补录与重新审核
- **PDA 负责**
  - 收货执行
  - 打印库存条码
  - 上架执行
  - 现场扫码与作业反馈

禁止同一业务动作在 ERP 与 PDA 两端重复实现两套主入口。

## 7. 打印中心修订

### 7.1 打印中心定位

打印中心不再视为辅助模块，而是一期核心能力。

### 7.2 打印中心必须覆盖的场景

- 收货标签
- 库存条码
- 塑料盒条码
- 物流条码
- 货架条码
- 产品标签
- 条码补打
- 跨网络打印

### 7.3 统一规则

- 所有业务打印统一进入打印任务中心
- 所有打印状态必须可追踪、可补打
- 业务页面不直接“本地打”，而是发送打印任务到指定打印机
- 打印日志必须可用于排错、补打与追责
- 收货订单详情必须能反查本单据关联打印记录
- 收货订单详情必须能按打印批次查看成功 / 失败 / 超时结果，并直接发起补打
- 打印失败与超时必须能回到异常工作台和原始业务单据
- 收货异常通知必须能直接回到原始收货详情的对应处理区，而不是只回模块列表
- 打印超时阈值必须走系统配置，不允许不同模块各自写死
- 收货主链补打必须支持整单、明细、条码三层粒度

---

## 8. 审计、异常与系统配置前移

以下能力视为一期 / 二期核心能力，而不是末尾收口项：

- 单据编号规则
- 操作日志
- 打印日志
- 异常工作台
- 提醒 / 通知
- 数据修复入口

原因：

- 当前系统已经高度依赖这些能力来排查问题、追踪打印、修复库存异常
- 如果这些能力不前移，主流程再多也不稳定

---

## 9. 当前阶段明确不做什么

以下能力在当前阶段明确不作为主线目标：

- 不做完整 CRM
- 不做完整财务总账系统
- 不做复杂审批流设计器
- 不做复杂消息规则设计器
- 不做多价格引擎并行
- 不做完整物流追踪运营平台
- 不做脱离现有骨架的新页面体系

### 作业绩效基础报表

作业绩效不是单页仪表盘，而是一组围绕仓库作业效率的基础报表页，当前先按“波次效率 + 仓库运营看板 + PDA 异常分析”三页落地。

#### 页面列表

- 波次效率报表：`/reports/wave-performance`
- 仓库运营看板：`/reports/warehouse-ops`
- PDA 异常分析：`/reports/pda-anomaly`

#### 数据口径

- 波次效率报表按筛选日期统计波次完成率、平均耗时、SKU 数、拣货量与作业效率
- 仓库运营看板按当日实时口径统计出库、拣货、入库、扫码、错误与撤销
- PDA 异常分析按筛选日期统计扫码错误、撤销、错误原因、异常条码与趋势
- 所有数据均以已上线业务数据为准，不新造独立统计口径

#### 回跳入口

- 波次效率报表的明细优先回跳波次详情或对应仓库任务
- 仓库运营看板的异常记录优先回跳仓库任务、扫描日志或收货相关详情
- PDA 异常分析的错误条目优先回跳异常工作台、对应条码记录或收货任务
- 若当前明细暂无更细入口，则至少回到对应报表中心的上一级页面

#### 验收标准

- 三页都能正常打开并通过报表烟雾检查
- 三页的筛选、排序、刷新和查询结果都能稳定返回
- 关键指标能和仓库作业实际数据一致
- 所有可回跳项都能落到真实存在的单据、任务或异常页
- 页面风格继续保持统一报表模板，不额外引入新骨架

## 14. 后续开发统一验收标准

蓝图改版后，所有新增模块或重构模块都按以下标准验收：

1. 是否复用统一页面骨架，而不是另做一套风格

4. 是否与当前 PDA、条码、仓库术语一致
5. 是否先上线最小闭环，再扩展高级能力
