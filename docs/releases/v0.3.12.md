# FlowCube v0.3.12

## 摘要

本版本聚焦**打印链路**：工作站级 SSE 订阅、一机多打物理路由、计费 SQL 兼容性，以及**货架页补打条码**；并增强未迁移库环境下的降级表现。

## 后端

- **打印任务 SSE**
  - 新增 `GET /api/print-jobs/listen/station`，请求头 `X-Client-Id` 与 `printers.client_id` 一致即可订阅本工作站全部绑定打印机任务（推荐）。
  - 保留 `GET /api/print-jobs/listen/:printerCode` 兼容旧客户端。
  - 创建任务后向「按编码订阅」与「按工作站订阅」的客户端去重推送。
  - `POST .../complete`、`.../fail` 支持 `X-Client-Id` 校验（与任务目标打印机的 `client_id` 一致）；仍支持 `X-Printer-Code`。
- **心跳**：`X-Client-Id` 与 body `clientId` 一致时，不再强制 `clientId` 以 `-打印机编码` 结尾。
- **`POST /api/printers/register-client`**：响应 `data.assigned[]` 含 `previousClientId`，便于发现多机抢绑同一逻辑打印机。
- **打印月度计费**：SQL 中对 `` `year_month` `` 使用反引号，避免部分 MySQL/MariaDB 下语法错误；告警扫描 SQL 同步修正。
- **租户打印策略缓存**：默认 TTL 30s → 15s（可用 `PRINT_TENANT_POLICY_CACHE_MS` 覆盖），策略变更更快生效。
- **打印机健康度**：`printer_health_stats` 缺失或字段异常时退化为「冷启动」健康度，避免 500。

## 打印客户端（`frontend/print-client.js`）

- 默认**工作站模式**：`--client-id` / `FLOWCUBE_CLIENT_ID`（默认 hostname），`--bind-code` 绑定多台逻辑打印机；仍可用 `--printer` 走旧版按编码订阅。
- **一机多打路由**：`--routes-file` 或 `FLOWCUBE_PRINT_ROUTES` / `FLOWCUBE_PRINT_ROUTES_FILE`，按打印机 `code` 配置 ZPL 的 `host`/`port`，或 `lp` / `winPrinter`（HTML）。
- 全局 **`FLOWCUBE_ZPL_HOST`** / **`--zpl-host`** 作为 ZPL 默认目标；未单独配置的 `code` 回退到该地址。
- 按 **`printerCode` 分队列串行**执行打印，同一斑马不乱序；ZPL 连接 30s 超时。
- 注册成功若发生**改绑**，控制台提示上一工作站 `client_id`。

## 前端

- **货架列表**：操作列增加「打印」，调用补打货架条码标签并入队（依赖后端绑定与打印客户端在线）。

## 升级与运维说明

1. **已有旧打印客户端**（`client_id` 形如 `主机名-编码`）若改用工单站模式，请使用 **`--client-id`** 与数据库一致，或重新注册使 `printers.client_id` 更新。
2. **一机多台斑马**：务必配置路由 JSON，或为每台保留独立 `code` 与 IP 映射。
3. **勿同时使用**同一任务的两种 SSE（`/listen/station` 与 `/listen/:code`）连到同一台机，以免重复出纸。
